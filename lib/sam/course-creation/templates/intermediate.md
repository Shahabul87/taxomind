## TAXOMIND INTERMEDIATE-LEVEL DESIGN PRINCIPLES

> An intermediate course is where intuition MEETS formalism. The learner already has the "feel" — now they need to see WHY things work, not just THAT they work. MIT's 18.06 (Linear Algebra with Gilbert Strang) is the gold standard: every theorem is motivated, visualized, and connected to something real.

### The 5 Pillars of a Taxomind Intermediate Course:

1. **Motivated Formalism** — No formula, framework, or technique appears without the learner first understanding WHY it needs to exist and WHAT problem it solves
2. **Visual Proofs & Geometric Intuition** — Every formal operation has a geometric or visual interpretation the learner can "see" in their mind
3. **Build-Then-Explain** — Learners implement or experience first (see it work), then unpack the theory behind it
4. **Failure-Driven Learning** — Show what goes wrong WITHOUT a technique, then introduce the technique as the rescue
5. **Connected Knowledge Graph** — Every concept explicitly connects to (a) the beginner-level intuition, (b) peer concepts in this level, and (c) what it unlocks at the advanced level

## CHAPTER DESIGN PATTERN

Each chapter should follow this arc:

- **Motivation**: Show a real problem or failure that this chapter's knowledge solves
- **Core Intuition Statement**: Link back to beginner-level understanding, then extend it
- **Lessons** (3-5 per chapter):
  - Each lesson: Motivation → Visual Intuition → Formal Definition → Geometric Interpretation → Implementation → Connection Map
- **Implementation Exercise**: Substantial hands-on exercise that builds real skill
- **"What Could Go Wrong?" Section**: Common failure modes and debugging strategies
- **Bridge to Next Chapter**: How this knowledge enables the next topic

### Example Chapter Pattern:

**Chapter N: [Topic] — [Formalism-Meets-Intuition Title]**
- Lesson N.1: Translating intuition to formalism (e.g., "From Analogy to Framework")
- Lesson N.2: The geometric/visual story (e.g., "The Geometry of [Concept]")
- Lesson N.3: Practical application (e.g., "[Technique] in Practice")
- Lesson N.4: Edge cases and failure modes (e.g., "When [Technique] Breaks")
- Implementation Exercise: Build something substantial
- Bridge: "We now understand X formally. But what happens when we need to handle Y?"

## INTERMEDIATE TEMPLATE STRUCTURAL RULES

### Content Rules
- **Every formal concept must be motivated** — "We need this because..." appears before any formula or framework
- **Every formal concept gets a visual interpretation** — if you can't draw it or diagram it, reconsider including it
- **Practice follows concept, never leads** — understand first, implement second
- **Derivations and reasoning shown step-by-step** with plain-language annotations between each step
- **Maximum 3 new formal concepts per lesson**
- **"Why, not just How" boxes** — every technique includes "Why was this invented?"

### Visual Rules
- Every module has 3-5 key diagrams, each serving a distinct purpose
- Geometric/spatial visualizations preferred over abstract diagrams
- Before/After comparisons for every technique (without vs with)
- Process flow diagrams for every multi-step operation

### Tone Rules
- "Let's figure this out together" — the learner is a peer
- Rigor without pretension
- Celebrate "aha moments" — mark them explicitly
- Reference beginner-level material naturally ("Remember when we...")

### Structure Rules
- Each module: 3-5 lessons
- Each lesson follows: Motivation → Visual Intuition → Formal Definition → Geometric Interpretation → Implementation → Connection Map
- Every module has a substantial practical exercise
- Every module ends with a "What Could Go Wrong?" section
- Course builds toward a capstone project integrating all skills

---

## QUALITY CHECKLIST FOR INTERMEDIATE COURSES

- Can a motivated self-learner with basic knowledge follow every explanation?
- Is every formal concept accompanied by a visual interpretation?
- Does the learner understand WHY each technique was invented?
- Are all exercises substantial enough to build real skill?
- Does each module explicitly connect to beginner-level intuitions?
- Would a second-year student say "I wish my course explained it like this"?
- Is the formalism rigorous but not gratuitously complex?
- Are common failure modes and debugging strategies covered?
- Does the course build toward genuine practical competency?
